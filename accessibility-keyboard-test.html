Accessibility Keyboard Navigation Test Guide

This document provides a checklist for manually testing keyboard accessibility of Shadcn UI components. Use this guide when the application is running to verify keyboard navigation works correctly.

How to Use This Guide
  1. Start with Tab key to navigate through interactive elements
  2. Verify focus is visibly indicated on each element
  3. Test element activation with Enter or Space keys
  4. Test arrow key navigation where applicable
  5. Verify Escape key closes popups, dialogs, and menus

Mark each test as passing or failing once completed.

Component Test Checklist

Form Controls

Input Fields

Test keyboard accessibility of input elements:

  1. Tab to focus on input fields
  2. Enter text into the input field
  3. Verify inputs with validation errors have descriptions readable by screen readers

Expected: Focus indicator should be visible around inputs, form validation errors should be associated with inputs.

Textarea Fields

Test keyboard accessibility of textarea elements:

  1. Tab to focus on textarea fields
  2. Enter text and use arrow keys to navigate within textarea
  3. Verify inputs with validation errors have descriptions readable by screen readers

Expected: Focus indicator should be visible around textareas, form validation errors should be associated with inputs.

Buttons

Test keyboard accessibility of buttons:

  1. Tab to focus on buttons
  2. Activate buttons with Enter or Space key
  3. Verify that disabled buttons are skipped in tab order

Expected: Focus indicator should be visible around buttons, and they should activate with Enter/Space.

Navigation Components

Navigation Menu

Test keyboard accessibility of navigation menus:

  1. Tab to focus on menu items
  2. Use arrow keys to navigate between menu items
  3. Use Enter to select or activate menu items
  4. Use Escape to close dropdown menus

Expected: Focus should be visible on menu items, and arrow keys should allow navigation between items.

Breadcrumbs

Test keyboard accessibility of breadcrumb navigation:

  1. Tab to focus on breadcrumb links
  2. Use Enter to activate breadcrumb links

Expected: Focus indicator should be visible on breadcrumb links.

Interactive Components

Accordion

Test keyboard accessibility of accordion components:

  1. Tab to focus on accordion headers
  2. Use Enter or Space to expand/collapse accordion panels
  3. Verify focus remains on the accordion header after expanding

Expected: Focus indicator should be visible on accordion headers, and they should expand/collapse with Enter/Space.

Dialog/Modal

Test keyboard accessibility of dialogs:

  1. Open a dialog and verify focus is trapped inside
  2. Tab through all interactive elements within the dialog
  3. Use Escape key to close the dialog
  4. Verify focus returns to the trigger element after dialog is closed

Expected: Focus should be contained within the dialog when open, and Escape should close it.

Carousel

Test keyboard accessibility of carousel components:

  1. Tab to focus on carousel navigation buttons
  2. Use arrow keys to navigate between carousel items
  3. Verify carousel auto-play stops when focused

Expected: Focus indicator should be visible on carousel controls, and arrow keys should navigate between slides.

ARIA Attribute Compliance

Component Required ARIA Implementation
Alert role="alert" ✓ Implemented with proper aria-labelledby
Dialog role="dialog", aria-modal="true" ✓ Implemented in Radix UI
Form fields aria-required, aria-invalid, aria-describedby ✓ Implemented in FormControl
Tabs role="tablist", role="tab", role="tabpanel" ✓ Implemented in Radix UI
Carousel role="region", aria-roledescription="carousel" ✓ Implemented with keyboard support

Focus Management

Important considerations for focus management:

  • Focus should be visually apparent on all interactive elements
  • Focus should be trapped within modal dialogs when open
  • Focus should return to trigger elements after modals close
  • Focus should not be lost during page navigation or content updates
  • Skip links should be available for keyboard users to bypass navigation

Motion and Animation

The codebase includes prefers-reduced-motion media query support in the CSS. This ensures that users who prefer reduced motion have animations disabled or simplified.

Example implementation in client/src/index.css:

@media (prefers-reduced-motion: reduce) {
  * {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
  
  .animate-pulse {
    animation: none !important;
  }
}
